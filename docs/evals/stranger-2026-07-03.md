**Unit:** release-docs/6 · **Method:** a fresh agent, given ONLY vibe's README
files (`README.md`, `spec/README.md`, `flow/README.md`, `docs/img/README.md`),
installed vibe into throwaway sandboxes and ran one quick-flow arc. It reported
each command tried, whether it worked, and every friction point. This is a
**gate**: blocking findings were fixed (or explicitly accepted) before the PR.

## Findings & disposition

| # | Sev | Area | Finding | Disposition |
|---|---|---|---|---|
| 1 | major | `flow/orders.sh` | On a fresh full install into a **non-git / non-`.spec`** target, `orders.sh` (the headline D12 feature) silently degraded to `state=unknown` for every skill-owning state — it required a `.git`/`.spec` root marker to locate the skills dir, which a fresh install lacks. `set-state.sh` worked, so transitions succeeded while orders silently broke. | **FIXED** — `orders.sh` now resolves the skills dir from its own script path (the skill's parent dir), falling back to marker search only for the source repo's canonical `flow/` path. Pinned by a regression test: a fresh non-git install resolves orders, byte-identical with/without a marker. |
| 2 | minor | root resolution divergence | `set-state.sh` and `orders.sh` resolved the repo differently, so transitions worked while orders didn't — surprising and undocumented. | **FIXED** — root cause of #1; the `orders.sh` self-location change removes the divergence. Regression test added. |
| 3 | minor | `spec/README.md` validator | `validate.sh` is cwd-relative; the README example used a relative path without stating cwd must be the project root. Run by absolute path elsewhere it silently validated a *different* `.spec/` and exited 0. | **FIXED** — README now says "run from your project root" and explains the cwd-relative resolution. |
| 4 | minor | `spec/README.md` bare setup | Quickstart listed only `/spec setup` (a Claude Code command); the bare-bash `setup.sh` was not in the "run directly" note, leaving non-Claude-Code users stuck. | **FIXED** — added `bash .agents/skills/spec/scripts/setup.sh` to the bare-user commands. |
| 5 | nit | `README.md` install prose | The install paragraph read as unconditional, but `--only spec` skips the AGENTS.md merge / cursor / plugin. | **FIXED** — scoped the paragraph to full install and noted what `--only spec` does. |
| 6 | nit | installer stdout | Completion output named `${CLAUDE_PROJECT_DIR}` while the READMEs/hooks use `${CLAUDE_PLUGIN_ROOT}`; not a bug (two different resolutions) but momentarily reads as a contradiction. | **FIXED** — installer output now distinguishes plugin-root (script loading) from project-dir (flow state). |

## Accepted / deferred (non-blocking)

- The plugin itself was not exercised end-to-end — it requires the interactive
  Claude Code `/plugin` command, which is not runnable from a shell. The manifest
  and hook wiring are covered by `tests/adapters/run.sh`. **Accepted** (documented
  Non-Goal: no live-session automation).
- `flow/reference/templates/AGENTS.md` still carries a legacy "Managed by
  vibe-setup" byline. Changing it requires a coordinated update to
  `merge-agents.sh`'s migration matcher + a re-merge; **deferred** to avoid churn
  right before release (tracked in REPORT.md).
