# Engineering Agent — Technical Architecture

## Design Philosophy

1. **File-based communication.** All state lives in `.spec/` files. No in-memory state survives session boundaries. Phases read files, not conversation history.
2. **Shell over code.** Prefer bash scripts over complex application code. The framework is glue, not a product.
3. **Hooks enforce, skills orchestrate.** PreToolUse hooks prevent writes in wrong phases. Skills handle the workflow logic.
4. **Subagents are disposable.** Each subagent gets a focused task + minimal context. They return compact summaries, not raw data.

### 3. Config Schema Design

The `.framework.json` needs a formal schema before implementation:

```json
{
  "$schema": "framework-config-v1",
  "version": 1,
  "phases": {
    "research": {
      "provider": "built-in",
      "config": {}
    },
    "discuss": {
      "provider": "built-in",
      "config": {}
    },
    "spec": {
      "provider": "spec"
    },
    "plan": {
      "provider": "built-in",
      "config": {
        "wave_grouping": true
      }
    },
    "implement": {
      "provider": "built-in",
      "config": {
        "tdd": false,
        "parallel_waves": true
      }
    },
    "review": {
      "provider": "simplify",
      "config": {}
    }
  },
  "patterns": {
    "wave_grouping": true,
    "plan_immutability": true,
    "gap_closure": true,
    "pressure_resistance": true
  },
  "installed_plugins": []
}
```

**Key design decisions for the schema:**
- `provider` is a single string, not an array (one provider per phase, except review which can chain)
- `patterns` section captures cross-cutting patterns that are on/off toggles
- `installed_plugins` is auto-populated by detect-plugins.sh
- `spec` phase has no `config` — it's always `/spec`, no options

### 4. Plugin Interface Contract

Each plugin adapter must implement:

```
Input:  phase name, phase config, path to .spec/
Output: writes results to .spec/ files, returns summary string

No plugin can:
- Skip a phase gate
- Write outside .spec/ during non-IMPLEMENT phases
- Auto-advance to the next phase
- Modify .framework.json
```

### 5. Script Dependencies

Scripts need these tools available:

| Tool | Used By | Purpose | Available? |
|------|---------|---------|------------|
| `bash` | All scripts | Script execution | ✅ Always |
| `cat` / `grep` / `sed` | detect-plugins.sh, phase-gate.sh | File reading, pattern matching | ✅ Always |
| `jq` | Config reading | Parse .framework.json | ⚠️ May need install |
| `ls` / `find` | detect-plugins.sh | Directory scanning | ✅ Always |

**Decision needed:** Do we require `jq` or parse JSON with bash/grep? `jq` is cleaner but adds a dependency. Bash-only parsing is uglier but zero-dep.

## Key Patterns

### Phase Routing Logic (Pseudocode)

```bash
# In /develop SKILL.md, each phase does:
route_phase() {
  local phase=$1
  local provider=$(read_config ".phases.$phase.provider")
  local config=$(read_config ".phases.$phase.config")

  case "$provider" in
    "built-in")
      # Execute built-in behavior for this phase
      run_builtin_$phase "$config"
      ;;
    *)
      # Check if plugin exists
      if skill_exists "$provider"; then
        # Delegate to plugin
        delegate_to_plugin "$provider" "$phase" "$config"
      else
        echo "WARN: $provider not installed, using built-in"
        run_builtin_$phase "$config"
      fi
      ;;
  esac
}
```

### Provider Mapping

| Phase | Provider | Delegates To |
|-------|----------|-------------|
| RESEARCH | `built-in` | Spawn parallel Explore agents |
| RESEARCH | `feature-dev` | Spawn code-explorer agents |
| RESEARCH | `superpowers` | Brainstorm + Explore |
| DISCUSS | `built-in` | AskUserQuestion structured prompts |
| DISCUSS | `superpowers` | `/superpowers:brainstorm` |
| SPEC | `spec` | Always `/spec` |
| PLAN | `built-in` | Wave-grouped plan writing |
| PLAN | `feature-dev` | Competing code-architect agents |
| IMPLEMENT | `built-in` | Wave-based subagent execution |
| REVIEW | `simplify` | `/simplify` multi-agent review |
| REVIEW | `feature-dev` | Code-reviewer with confidence scoring |

### Built-in Patterns (Always Active)

| Pattern | Enforcement | Source |
|---------|------------|--------|
| Wave grouping | Plan structure | GSD |
| Plan immutability | Hook blocks plan edits during IMPLEMENT unless gap-closure triggered | GSD |
| Gap closure | IMPLEMENT detects reality ≠ plan → pauses → updates plan → continues | GSD |
| Pressure resistance | SKILL.md instructions resist "just code it" | Superpowers |
| Phase gates | PreToolUse hooks | Superpowers |
| File-based state | All output to `.spec/` | Ours |
| Fresh subagents | Each task = new agent with minimal context | GSD |

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Plugin skill interfaces change | Pin to known patterns, validate at setup |
| Config stale (plugin uninstalled) | Validate at `/develop` startup, warn + fallback |
| Subagent context overflow | Compact summaries only (~95% reduction) |
| `jq` not installed | Either require it or provide bash-only JSON parsing fallback |
| Framework overhead discourages adoption | Zero-plugin setup works instantly, overhead is near-zero |
